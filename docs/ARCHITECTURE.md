# Архитектура проекта

Calendar Sync Service построен вокруг общей внутренней модели события,
канонического состояния в SQLite и сменных календарных адаптеров.

## Участники синхронизации

В live-режиме используются четыре календаря:

- `developer_1` - Yandex Calendar через CalDAV;
- `developer_2` - Google Calendar API;
- `manager` - локальный JSON mock для Outlook Calendar;
- `leader` - второй Yandex Calendar через CalDAV.

Outlook оставлен JSON mock-адаптером, потому что для тестового прототипа это
достаточно показывает работу с третьей экосистемой без отдельной регистрации
Microsoft приложения.

## Модель события

Все внешние события приводятся к `CalendarEvent`:

- `id`;
- `title`;
- `description`;
- `start_time`;
- `end_time`;
- `organizer`;
- `attendees`;
- `source_system`;
- `source_owner`;
- `status`;
- `updated_at`.

## Хранилище

SQLite хранит три таблицы:

- `synced_events` - каноническое состояние события по `sync_group_id`;
- `event_mappings` - соответствия `sync_group_id` и event id в каждом календаре;
- `sync_logs` - журнал операций синхронизации.

`synced_events` является источником истины. Сервис сначала импортирует изменения
из календарей в эту таблицу, затем проецирует каноническое состояние обратно во
все адаптеры.

## Защита от дублей и циклов

Для всех копий одного события используется общий `sync_group_id`. Он хранится в
SQLite и дополнительно записывается в описание события:

```text
[SYNC_ID: sync_xxx]
[SOURCE: google]
```

Если сервис видит уже размеченную копию, он восстанавливает mapping и не создает
дубликат.

## Логика синхронизации

1. Сервис читает события из всех адаптеров.
2. Для каждого события находит или создает `sync_group_id`.
3. Входящие создания и обновления записываются в `synced_events`.
4. Если mapped-событие исчезло из календаря или пришло как terminal event,
   canonical event помечается удаленным.
5. Каноническое состояние проецируется во все календари.
6. Отсутствующие активные копии создаются.
7. Устаревшие копии обновляются.
8. Удаленные группы удаляются из календарей через `delete_event`.
9. В SQLite остается tombstone, чтобы событие не было создано заново.

## Основные команды проверки

Автоматическая live-проверка:

```powershell
python main.py live-demo
```

Ручная проверка:

```powershell
python main.py watch
python main.py live-links
```

`watch` раз в 10 секунд проверяет изменения. `live-links` печатает ссылки на
тестовые календари и пути к JSON mock-файлам.

## Подключение внешних календарей

Yandex-аккаунты подключаются через CalDAV по логину и паролю приложения из
локального `.env`. Для смены Yandex пользователя нужно заменить переменные в
`.env`, удалить соответствующие `data/yandex*_calendar_config.json` и заново
создать sync-календарь.

Google подключается через OAuth-токен Calendar API. Другой Gmail не добавляется
через `.env`: пользователь либо сам проходит OAuth и получает свой
`data/google_token.json`, либо владелец текущего Google календаря просто делится
календарем с этим Gmail с правом редактирования событий.
